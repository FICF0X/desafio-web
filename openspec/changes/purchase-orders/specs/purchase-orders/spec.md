# Purchase Orders Specification

## Purpose

Defines the behavioral contract for two new capabilities delivered together:
`suppliers` (minimal entity supporting PO selection) and `purchase-orders`
(PO header + line items lifecycle, listing, and detail view). This is Module 3
of the comercial management system.

## Scope

### In Scope
- Supplier list and create (minimal entity; name required).
- PO create: supplier selection, line items (product + quantity + unit cost), atomic persist.
- PO list view and PO detail view.
- PO cancellation (pending → cancelled).
- Status values reserved for Module 4 goods receipt (pending → received).
- Supabase RLS on all three tables restricted to `authenticated` role.

### Out of Scope
- PO edit or delete after creation.
- PO approval workflow.
- Partial receipts (Module 4 in-scope, not here).
- PDF export.
- Full supplier CRUD (edit, delete, deactivate).
- RBAC / per-user ownership of POs.

---

## Requirements

### Requirement: Create Supplier

The system MUST allow creating a supplier with `name` as the only required field.
`tax_id` (RUC), `email`, and `phone` fields are optional. The supplier MUST be
persisted to the `suppliers` table and MUST immediately be selectable in the PO
create form.

#### Scenario: Happy path — supplier created

- GIVEN the authenticated user is on the Create Supplier page (`/suppliers/new`)
- WHEN they enter a non-empty `name` and submit the form
- THEN the supplier MUST be persisted to the `suppliers` table
- AND the user MUST be redirected to the supplier list
- AND the new supplier MUST appear in the list

#### Scenario: Missing name

- GIVEN the authenticated user is on the Create Supplier page
- WHEN they submit the form with `name` left empty
- THEN an inline validation error MUST appear on the `name` field
- AND the form MUST NOT submit

#### Scenario: Optional fields omitted

- GIVEN the authenticated user submits a supplier form with only `name` filled
- WHEN the supplier is persisted
- THEN `tax_id`, `email`, and `phone` MUST be `null` in the database
- AND no error SHALL occur

---

### Requirement: List Suppliers

The system MUST display all suppliers in a table. Seeded sample suppliers MUST be
present after migration. The list MUST show at minimum: supplier name and tax_id.

#### Scenario: Populated list

- GIVEN at least one supplier (or seeded sample suppliers) exists
- WHEN the authenticated user navigates to `/suppliers`
- THEN the supplier table MUST render with at least columns: Name, Tax ID
- AND each row MUST correspond to a `suppliers` record

#### Scenario: Empty state

- GIVEN no suppliers have been created and seeds have not been applied
- WHEN the list renders
- THEN an empty-state message MUST be displayed
- AND the table MUST NOT show blank rows

---

### Requirement: Create Purchase Order

The system MUST create a PO atomically (header + all line items in a single
transaction — all succeed or all fail). A PO MUST NOT be created without a
supplier. A PO MUST NOT be created without at least one line item. The PO header
MUST receive a unique code in format `OC-YYYY-NNNN` (generated by the Server
Action via per-year sequential count). `status` MUST default to `pending`. The
stored `total` on the header MUST equal the sum of all line-item subtotals.

Duplicate product lines are NOT allowed: each product MUST appear at most once
per PO. The form MUST prevent adding the same product twice.

#### Scenario: Happy path — PO created with multiple items

- GIVEN the authenticated user is on the Create PO page
- AND at least one supplier exists and at least two products exist
- WHEN they select a supplier, add two distinct line items (each with quantity > 0 and unit_cost ≥ 0), and submit
- THEN the PO header MUST be inserted into `purchase_orders` with status `pending` and code matching `OC-YYYY-NNNN`
- AND each line item MUST be inserted into `purchase_order_items` with correct `subtotal = quantity × unit_cost`
- AND the header `total` MUST equal the sum of all line-item subtotals
- AND the user MUST be redirected to the PO detail page

#### Scenario: Reject PO with no supplier

- GIVEN the authenticated user is on the Create PO page
- WHEN they submit without selecting a supplier
- THEN an inline validation error MUST appear on the supplier field
- AND no database row SHALL be inserted

#### Scenario: Reject empty PO (no line items)

- GIVEN the authenticated user is on the Create PO page with a supplier selected
- WHEN they submit without adding any line item
- THEN the form MUST display an error indicating at least one line item is required
- AND no database row SHALL be inserted

#### Scenario: Reject line item with quantity ≤ 0

- GIVEN the user adds a line item and sets `quantity` to 0 or a negative value
- WHEN they attempt to submit
- THEN an inline validation error MUST appear on the `quantity` field
- AND the form MUST NOT submit

#### Scenario: Reject duplicate product in same PO

- GIVEN the user has already added product `P` as a line item
- WHEN they attempt to add product `P` again
- THEN the form MUST prevent the addition (disable the product in the picker or display an error)
- AND only one line for product `P` MUST exist when the PO is submitted

#### Scenario: Atomic persistence — partial failure rolls back

- GIVEN the Server Action begins inserting a PO header and items
- WHEN any item insert fails (e.g. FK violation)
- THEN the header insert MUST also be rolled back
- AND no partial PO record SHALL exist in the database

#### Scenario: Generated PO code is unique per year

- GIVEN PO `OC-2026-0001` already exists for year 2026
- WHEN a new PO is created in year 2026
- THEN the new code MUST be `OC-2026-0002`
- AND the `code` column UNIQUE constraint MUST prevent any collision

---

### Requirement: Line Item Rules

Each purchase order item MUST satisfy: `quantity` is a positive integer (`> 0`);
`unit_cost` is a non-negative numeric (`≥ 0`); `subtotal` MUST equal
`quantity × unit_cost` and MUST be computed and stored by the Server Action.
A zero `unit_cost` is valid (e.g. promotional items).

#### Scenario: Valid line item — unit_cost zero

- GIVEN the user adds a line item with `quantity = 2` and `unit_cost = 0`
- WHEN the PO is created
- THEN `subtotal` MUST be `0` and the line item MUST be persisted
- AND the header `total` MUST include this subtotal (contributing 0)

#### Scenario: Non-integer quantity rejected

- GIVEN the user types a decimal value (e.g. `1.5`) for `quantity`
- WHEN they attempt to submit
- THEN an inline validation error MUST appear stating quantity must be a whole number
- AND the form MUST NOT submit

---

### Requirement: List Purchase Orders

The system MUST display all purchase orders in a table ordered by most recent
(`order_date` descending, then `created_at` descending). Required columns: Code,
Supplier Name, Order Date, Status, Total.

#### Scenario: Populated list

- GIVEN at least one PO exists
- WHEN the authenticated user navigates to `/purchase-orders`
- THEN the table MUST render rows with columns: Code, Supplier Name, Order Date, Status, Total
- AND rows MUST be ordered most-recent first

#### Scenario: Empty state

- GIVEN no POs have been created
- WHEN the list renders
- THEN an empty-state message MUST be displayed
- AND no blank rows SHALL appear

---

### Requirement: View Purchase Order Detail

The system MUST display a PO detail page with: header info (code, order_date,
status, notes, total), supplier name, and a table of line items (product name,
quantity, unit_cost, subtotal).

#### Scenario: Happy path — detail renders

- GIVEN a PO with code `OC-2026-0001` and two line items exists
- WHEN the authenticated user navigates to `/purchase-orders/[id]`
- THEN the header fields (code, status, date, total) MUST be rendered
- AND the supplier name MUST be shown
- AND each line item MUST appear with product name, quantity, unit_cost, and subtotal

#### Scenario: PO not found

- GIVEN the user navigates to a PO ID that does not exist
- WHEN the server resolves the route
- THEN the page MUST display a not-found state or redirect to the list
- AND no unhandled exception SHALL reach the client

---

### Requirement: PO Status Lifecycle

The `status` field uses an enum with values `pending`, `received`, `cancelled`.
A `pending` PO MAY be cancelled by the authenticated user. A `received` PO MUST
NOT be cancelled or modified. Transitioning from `pending` to `received` is
OUT OF SCOPE for this change — it is reserved for Module 4 (goods receipt).

#### Scenario: Cancel a pending PO

- GIVEN a PO with `status = pending` exists
- WHEN the authenticated user triggers the Cancel action on that PO
- THEN `status` MUST be set to `cancelled` in the `purchase_orders` table
- AND the PO list and detail MUST reflect the updated status

#### Scenario: Cannot cancel a received PO

- GIVEN a PO with `status = received` exists (set by Module 4)
- WHEN any user attempts to trigger a cancel or edit action on it
- THEN the system MUST reject the operation (return an error or disable the control)
- AND `status` MUST remain `received`

#### Scenario: Status pending → received is reserved for Module 4

- GIVEN a PO with `status = pending`
- WHEN Module 4 performs a goods receipt operation referencing this PO
- THEN `status` MAY be transitioned to `received` by Module 4's Server Action (not by Module 3 UI)
- AND this transition is OUT OF SCOPE for this spec

---

### Requirement: Edit Policy

A PO MUST NOT be editable after creation in this iteration. No edit form or
update Server Action SHALL exist for `purchase_orders` or `purchase_order_items`
in Module 3. This keeps scope tight and avoids total-recompute and audit concerns.

#### Scenario: No edit UI is exposed

- GIVEN a PO of any status exists
- WHEN the authenticated user views the PO list or detail
- THEN no "Edit" button or edit route SHALL be available for the PO header or its items
- AND no Server Action for updating a PO MUST exist in this feature scope

---

### Requirement: Access Control (RLS + Route Guard)

ALL operations on `suppliers`, `purchase_orders`, and `purchase_order_items` MUST
require an authenticated session. Defense is two-layered: (1) the `(app)` route
group layout guard / middleware (`proxy.ts`) redirects unauthenticated users to
`/login`; (2) Supabase RLS policies on all three tables restrict `SELECT`,
`INSERT`, and `UPDATE` to the `authenticated` role. Unauthenticated access MUST
be blocked at both layers.

#### Scenario: Unauthenticated access is redirected

- GIVEN a user has no valid Supabase session cookie
- WHEN they navigate directly to `/purchase-orders` or `/suppliers`
- THEN `proxy.ts` middleware MUST redirect them to `/login` before the page renders

#### Scenario: Unauthenticated DB access is blocked by RLS

- GIVEN an HTTP request reaches Supabase using the `anon` role
- WHEN it attempts `SELECT`, `INSERT`, or `UPDATE` on `suppliers`, `purchase_orders`, or `purchase_order_items`
- THEN Supabase RLS MUST deny the operation
- AND no data SHALL be returned or mutated

#### Scenario: Authenticated user can perform all in-scope operations

- GIVEN a user has a valid `authenticated` Supabase session
- WHEN they create a supplier, create a PO, list POs, view PO detail, or cancel a pending PO
- THEN RLS MUST permit all operations
- AND each operation MUST complete successfully subject to business-rule constraints above

---

### Requirement: Referential Integrity

Products referenced in PO line items MUST use soft-delete only (`is_active` flag).
Products MUST NOT be hard-deleted from the `products` table, ensuring PO history
remains intact and product FKs in `purchase_order_items` are never orphaned.
Deleting a PO header MUST cascade-delete all its line items.

#### Scenario: Soft-deleted product still visible in PO history

- GIVEN a product `P` was used in a PO line item and later soft-deleted (`is_active = false`)
- WHEN the user views the PO detail
- THEN the line item row MUST still render (product name MAY show as inactive but MUST be readable)
- AND the `product_id` FK MUST still resolve in the database

#### Scenario: PO header delete cascades to items

- GIVEN a PO with three line items exists
- WHEN the PO header row is deleted (e.g. during a rollback/test cleanup)
- THEN all three line items MUST be deleted automatically via `ON DELETE CASCADE`
- AND no orphaned `purchase_order_items` rows SHALL remain
